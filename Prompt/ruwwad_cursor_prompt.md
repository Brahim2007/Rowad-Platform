## 0) الدور والسياق (System Context)

أنت مهندس Full-Stack أول متخصّص في تطبيقات **عربية RTL** عالية الجودة. مهمتك بناء **«لوحة أثر الرواد»** — نظام ويب لقياس وتتبّع أثر أعضاء **شبكة الرواد الإلكترونية** عبر المنصّات، يحوّل نشاط كل عضو إلى نقاط ومستويات ودروع ومكافآت، مع لوحات متابعة وتقارير قابلة للطباعة.

يوجد **نموذج أولي (prototype)** يعمل على `localStorage` كملف HTML واحد. هدفك إعادة بنائه كنظام **إنتاجي متعدّد المستخدمين** بقاعدة بيانات حقيقية ومصادقة وصلاحيات، **مع الحفاظ الكامل والحرفي على منطق احتساب النقاط والمستويات والمكافآت** الموصوف أدناه. لا تبتكر قواعد جديدة للأعمال؛ التزم بالأرقام والعتبات كما هي.

**مبادئ ملزمة:**
- كل النصوص الظاهرة بالعربية، والاتجاه `dir="rtl"` افتراضيًا.
- نوع آمن صارم: TypeScript فقط، بلا `any` إلا لضرورة موثّقة.
- لا تكتب أكثر من ملف/مكوّن قبل أن يُجمَّع المشروع (`build` ناجح).
- اشرح أي قرار معماري في تعليق مختصر أعلى الملف.
- لا تستخدم `localStorage` كمصدر حقيقة؛ هو فقط لتفضيلات الواجهة (الثيم، آخر فلتر).

## 2) هوية النظام والمصطلحات (Domain Glossary)

- **شبكة الرواد الإلكترونية**: الكيان الجامع. تحته **منصّات (Platforms)** وأعضاء.
- **المحاور الأربعة (Categories):**
  1. `النشاط الرقمي`
  2. `المشاركة العلمية والفعاليات`
  3. `المبادرات والإنتاج`
  4. `الالتزام والانضباط`
- **الصفات (Roles داخل الشبكة، ليست أدوار النظام):** `باحث ومفكر`، `مؤثر رقمي`، `متطوع`، `مشرف`، `رئيس منصة`.
- **حالة النشاط (Approval Status):** `معتمد`، `قيد المراجعة`، `مرفوض`.
- **حالة العضو (Member Status):** `نشط`، `متوقف`، `تحت التجربة`.
- **النقاط → المستوى → الاستحقاق/المكافأة → الدروع → بوابة العمرة**.

## 3) نموذج البيانات وقاعدة البيانات (Supabase Schema)

أنشئ مخطّط Postgres التالي عبر ملف هجرة SQL في `supabase/migrations/`. استخدم `uuid` كمفاتيح، و`timestamptz` للتواريخ، وفعّل **RLS** على كل الجداول.

```sql
-- المنصّات
create table platforms (
  id uuid primary key default gen_random_uuid(),
  name text not null unique,
  status text not null default 'نشط',          -- نشط | متوقف | تحت التجربة
  created_at timestamptz default now()
);

-- ملفات المستخدمين (مرتبطة بـ auth.users) — أدوار النظام
create table profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  full_name text,
  app_role text not null default 'member',      -- admin | supervisor | member
  member_id uuid references members(id),         -- ربط حساب الدخول بعضو الشبكة (اختياري)
  created_at timestamptz default now()
);

-- الأعضاء
create table members (
  id uuid primary key default gen_random_uuid(),
  name text not null,
  network_role text not null,                    -- الصفة: باحث ومفكر... إلخ
  platform_id uuid references platforms(id),
  member_status text not null default 'نشط',
  join_date date,
  phone text,
  email text,
  note text,
  created_at timestamptz default now()
);

-- كتالوج الأنشطة (قابل للتعديل من الإعدادات)
create table activity_types (
  id uuid primary key default gen_random_uuid(),
  name text not null,
  points int not null,                           -- قد تكون سالبة (خصومات)
  category text not null,                         -- أحد المحاور الأربعة
  note text default ''
);

-- سجلّ الأنشطة المُدخلة
create table activities (
  id uuid primary key default gen_random_uuid(),
  member_id uuid not null references members(id) on delete cascade,
  activity_type_id uuid not null references activity_types(id),
  count numeric not null default 1,
  quality text not null default 'مقبول',         -- ضعيف|مقبول|جيد|ممتاز|استثنائي
  status text not null default 'قيد المراجعة',   -- معتمد|قيد المراجعة|مرفوض
  date date not null,
  link text,
  note text,
  created_by uuid references auth.users(id),
  created_at timestamptz default now()
);

-- الدروع والمكافآت
create table awards (
  id uuid primary key default gen_random_uuid(),
  member_id uuid not null references members(id) on delete cascade,
  type text not null,                            -- درع | مكافأة
  title text not null,                            -- من كتالوج الدروع أو وصف المكافأة
  value numeric default 0,                        -- قيمة المكافأة المالية إن وُجدت
  date date default current_date,
  note text
);

-- بوابات المهام الإلزامية الشهرية (member × سنة × شهر)
create table monthly_gates (
  id uuid primary key default gen_random_uuid(),
  member_id uuid not null references members(id) on delete cascade,
  year int not null,
  month int not null,
  passed boolean not null default true,
  unique (member_id, year, month)
);

-- إعدادات النظام (صف مفرد JSON) — المستويات، الجودة، شرائح المكافأة، شروط العمرة
create table app_settings (
  id int primary key default 1,
  quality_bonus jsonb not null,
  levels jsonb not null,
  reward_tiers jsonb not null,
  umrah jsonb not null
);
```

## 4) محرّك احتساب النقاط (Scoring Engine) — التزم به حرفيًا

ضع كل المنطق في وحدة نقية `lib/scoring.ts` (دوال خالصة قابلة للاختبار):

1. **نقاط نشاط واحد (`finalPoints`)**:
   - إذا كانت الحالة `مرفوض` أو `قيد المراجعة` → **النقاط = 0**.
   - وإلا: `النقاط = (count × points الخاص بنوع النشاط) + bonus الجودة`.
   - `count` الفارغ يُعامل كـ 1.

2. **إجمالي العضو (`memberTotal`)** = مجموع `finalPoints` لكل أنشطته.

3. **النقاط حسب المحور (`memberByCat`)** = تجميع `finalPoints` حسب محور نوع النشاط.

4. **النطاق الزمني (`scoped`)**: دعم ثلاثة نطاقات: `week` (آخر 7 أيام منتهية بتاريخ مرجعي)، `month` (سنة+شهر)، `all`. أي حساب نقاط/أنشطة يجب أن يقبل نطاقًا.

5. **المستوى (`levelOf`)**: أول مستوى تقع نقطته بين `from..to`، وإلا آخر مستوى.

6. **حالة نشاط العضو (`activityStatus`)** بناءً على عدد الأيام منذ **آخر نشاط معتمد**:
   - ≤ 7 يوم → `نشط هذا الأسبوع` (أخضر)
   - ≤ 30 → `نشط هذا الشهر` (أخضر)
   - ≤ 60 → `خامل (٣٠–٦٠ يوم)` (برتقالي)
   - > 60 → `متوقف (+٦٠ يوم)` / `لم يبدأ بعد` (أحمر)

7. **المكافأة الشهرية**: تُحسب من شرائح المكافأة على **نقاط الشهر**، **لكن** تُمنح فقط إذا كانت **بوابة المهام الإلزامية** لذلك الشهر `passed = true`؛ وإلا تُعرض «لا استحقاق».

8. **أهلية العمرة (`umrahEligible`)** لعضو في سنة معيّنة: تتحقق كل الشروط معًا:
   - نقاط السنة ≥ `minYearly` (3000)
   - عدد الأشهر التي بها نشاط ≥ `minMonths` (9)
   - عدد المبادرات (أنشطة يحتوي اسمها على «مبادرة») ≥ `minInitiatives` (1)
   - يوجد نشاط واحد على الأقل بجودة `ممتاز` أو `استثنائي` (`requireExcellent`)
   - لا توجد أي مخالفات/خصومات (نقاط نوع النشاط سالبة) في تلك السنة (`noViolations`)

9. **تجميع المنصّات (`platformAgg`)**: لكل منصّة احسب عدد الأعضاء، عدد الأنشطة، مجموع النقاط، أفضل عضو، المتوسط — مرتّبة تنازليًا بالنقاط.

> اكتب **اختبارات Vitest** لكل دالة من الدوال أعلاه بحالات حدّية (نشاط مرفوض، عدّاد فارغ، خصومات، حدود المستويات، تحقق العمرة بكل شرط مفقود على حدة).

## 5) المصادقة والأدوار والصلاحيات (Auth + RBAC + RLS)

ثلاثة أدوار نظام في `profiles.app_role`:

| الدور | الصلاحيات |
|---|---|
| **admin (إدارة)** | كل شيء: إضافة/تعديل/حذف الأعضاء والأنشطة، اعتماد/رفض، تعديل الإعدادات والكتالوج، منح الدروع والمكافآت، إدارة البوابات، النسخ الاحتياطي. |
| **supervisor (مشرف)** | إضافة/تعديل الأعضاء والأنشطة، **لا يحذف الأعضاء**، لا يعدّل إعدادات النظام. |
| **member (عضو)** | قراءة فقط: يرى لوحته وبطاقته وترتيبه؛ **لا يعدّل** شيئًا. |

- استخدم **Supabase Auth** (Email/Password + Magic Link). أنشئ trigger ينشئ صفًّا في `profiles` عند التسجيل.
- طبّق **RLS policies** تعكس الجدول أعلاه (مثلًا: العضو يقرأ كل شيء لكن لا `insert/update/delete`؛ المشرف لا يستطيع `delete` على `members`؛ تعديل `app_settings` و`activity_types` للـ admin فقط).
- في الواجهة: أخفِ أزرار الإضافة/التعديل/الحذف حسب الدور (مكوّن `<Can role="admin">`), لكن **اعتمد على RLS كحارس حقيقي** لا على الإخفاء فقط.

## 6) الصفحات والمسارات (App Router Pages)

لوحة محمية تحت `(dashboard)/` مع تبويب علوي بنفس ترتيب النموذج الأصلي:

| المسار | العنوان | المحتوى |
|---|---|---|
| `/dashboard` | الرئيسية | KPIs (إجمالي الأعضاء/الأنشطة/النقاط/المنصّات)، فائزو الفترة، **رسم أعلى 8 أعضاء** (شريطي)، **توزيع الأنشطة على المحاور** (دائري)، جدول ترتيب الأعضاء + فلتر بالنطاق (أسبوع/شهر/الكل). |
| `/members` | الأعضاء | جدول الأعضاء (الاسم، الصفة، المنصّة، تاريخ الانضمام، تواصل)، بحث/فلترة، نموذج إضافة/تعديل، حذف (admin). |
| `/activities` | الأنشطة | سجلّ الأنشطة، فلترة بالعضو/المحور/الحالة/التاريخ، تسجيل نشاط، **سير اعتماد** (معتمد/قيد المراجعة/مرفوض) للمشرف فالإدارة. |
| `/pulse` | المتابعة الدورية | اتجاه نشاط الشبكة (آخر 8 أسابيع / 6 أشهر)، نبض المنصّات، أعضاء يحتاجون متابعة (خامل/متوقف)، الأكثر تفاعلًا مؤخرًا. |
| `/card/[memberId]` | بطاقة الرائد | بطاقة عضو كاملة: المستوى، شريط التقدّم للمستوى التالي، **رادار بالمحاور الأربعة**، **اتجاه شهري**، الدروع، أبرز الأعمال (مع روابط)، **زر طباعة**. |
| `/compare` | المقارنة | مقارنة حتى 4 أعضاء جنبًا إلى جنب عبر معايير (النقاط، المستوى، النشاط، المحاور، الدروع). |
| `/rewards` | المكافآت | **حاسبة الاستحقاق الشهري** (نقاط الشهر → شريحة، مشروطة بالبوابة)، إدارة البوابات الإلزامية، سجلّ الدروع والمكافآت، منحها (admin). |
| `/reports` | التقارير | مولّد تقارير **أسبوعي/شهري/سنوي**، فلترة بمنصّة/صفة، عرض مستند قابل للطباعة (`@media print` → PDF). |
| `/settings` | الإعدادات | تحرير كتالوج الأنشطة، بونص الجودة، المستويات، شرائح المكافأة، شروط العمرة، إدارة المنصّات (admin فقط). |

**رأس الصفحة (Header):** شعار + اسم الشبكة، جرس تنبيهات (عدّاد + قائمة منبثقة)، زر الدليل، قائمة أدوات الإدارة (نسخ احتياطي JSON / استرجاع / إعادة تعيين — للـ admin)، مبدّل الدور للعرض (للتجربة فقط في بيئة التطوير).

**التنبيهات (الجرس):** ولّدها برمجيًا — أمثلة: «بوابة متعثّرة: فلان لديه نقاط هذا الشهر دون استحقاق»، «عضو متوقف +60 يوم»، «أنشطة قيد المراجعة بانتظار الاعتماد».

- **الترويسة:** تدرّج navy → navy-2، علامة ذهبية مربّعة الزوايا.
- **البطاقات (panels):** خلفية بيضاء، ظل خفيف، `--radius-lg`، عناوين بلون navy وأيقونة Phosphor.
- **الشارات (badges):** كبسولات مدوّرة؛ لون حسب المحور/الحالة.
- **شريط الحفظ:** إشعار «تم الحفظ» عائم بعد كل عملية كتابة ناجحة.
- **الطباعة:** `@media print` يُخفي عناصر `.no-print` ويعرض المستند فقط بحواف نظيفة.
- اتبع إرشادات الملف `frontend-design` إن كان متاحًا (tokens, typography, spacing) لتجنّب المظهر القالبي الافتراضي.

## 8) النماذج والتحقق (Forms & Validation)

- استخدم `react-hook-form` + `zod` لكل نموذج (عضو، نشاط، درع/مكافأة، إعدادات).
- مخطّط نشاط مثال:
  ```ts
  const activitySchema = z.object({
    memberId: z.string().uuid(),
    activityTypeId: z.string().uuid(),
    count: z.coerce.number().min(0).default(1),
    quality: z.enum(["ضعيف","مقبول","جيد","ممتاز","استثنائي"]),
    status: z.enum(["معتمد","قيد المراجعة","مرفوض"]),
    date: z.coerce.date(),
    link: z.string().url().optional().or(z.literal("")),
    note: z.string().optional(),
  });
  ```
- أظهر **النقاط المحسوبة لحظيًا** أسفل نموذج النشاط (count × points + bonus) ليرى المُدخِل الأثر قبل الحفظ.
- كل الكتابة عبر **Server Actions** تتحقق من الدور على الخادم قبل التنفيذ.

## 11) الترحيل من النموذج الأولي (Migration)

النموذج الحالي يخزّن كل شيء في مفتاح `localStorage` اسمه `ruwad_impact_db_v3` بشكل JSON يحوي: `members, activities, awards, criteria{activities,quality,levels}, settings{rewardTiers,umrah}, gates, ui`.

اكتب **سكربت استيراد** `scripts/import-legacy.ts` يقرأ ملف JSON مُصدَّرًا من النموذج القديم ويحقنه في Supabase: ينشئ المنصّات من قيم `member.platform` الفريدة، يربط الأعضاء بها، يحوّل أسماء الأنشطة إلى `activity_type_id`، ويحافظ على التواريخ والحالات والدروع والبوابات. ووفّر **زر استرجاع JSON** في أدوات الإدارة يستدعي هذا المسار.

## 13) مراحل التنفيذ (Build Order — التزم بالترتيب)

1. **التأسيس:** Next.js + TS + Tailwind + shadcn + الخط العربي + تخطيط RTL + الثيم.
2. **قاعدة البيانات:** هجرات Supabase + RLS + بيانات seed (المنصّات، الكتالوج، الإعدادات).
3. **المصادقة والأدوار:** تسجيل/دخول + profiles + مكوّن `Can` + حراسة المسارات.
4. **محرّك النقاط:** `lib/scoring.ts` + اختبارات Vitest كاملة (لا تتقدّم قبل نجاحها).
5. **الأعضاء + الأنشطة:** CRUD + سير الاعتماد + النقاط اللحظية.
6. **الرئيسية + الرسوم:** KPIs + الرسمان + جدول الترتيب + الفلترة بالنطاق.
7. **بطاقة الرائد + المقارنة.**
8. **المتابعة الدورية + التنبيهات (الجرس).**
9. **المكافآت + البوابات + الدروع + بوابة العمرة.**
10. **التقارير + تصدير CSV + الطباعة.**
11. **الإعدادات (تحرير الكتالوج/المستويات/الشرائح/العمرة).**
12. **الترحيل + النسخ الاحتياطي/الاسترجاع + التلميع وإمكانية الوصول.**
