### المرحلة 5 — الأرشيف والوثائق 🟡

> الهدف: تحويل قسم المكتبة المعرفية إلى أرشيف مؤسسي قابل للبحث مع تتبع كامل للتعديلات.

**Schema الجديد:**
```
Document {
  id, title, type, platformId?, periodYear, periodMonth?
  fileUrl, fileType, fileSize
  uploadedBy (AdminUser/PlatformManager), uploadedAt
  lastEditedBy, lastEditedAt
  status: DRAFT | APPROVED | ARCHIVED
  description?, tags?
}

DocumentVersion {
  id, documentId, version, fileUrl
  editedBy, editedAt, changeNote
}
```

**أنواع الوثائق المقترحة:**
`تقرير` · `ميزانية` · `محضر اجتماع` · `خطة عمل` · `إعلان` · `نشرة` · `أخرى`

**الصفحات الجديدة:**
- [x] `/admin/documents` — الأرشيف الرئيسي:
  - **تبويب "كل الوثائق"**: جدول مع فلتر بالنوع، المنصة، الفترة، الحالة. يظهر لكل وثيقة: العنوان، النوع، المنصة، الرافع، تاريخ الرفع، آخر تعديل.
  - **تبويب "بحث"**: بحث نصي فوري في العنوان والوصف والوسوم.
  - **تبويب "رفع وثيقة"**: نموذج رفع مع تصنيف كامل.
  - **تبويب "سجل التعديلات"**: جدول زمني لكل من رفع أو عدّل وثيقة.
- [x] API: `GET /api/admin/documents` مع فلتر متكامل.
- [x] API: `POST /api/admin/documents` — رفع وثيقة جديدة.
- [x] API: `PUT /api/admin/documents/[id]` — تعديل مع تسجيل نسخة سابقة.
- [x] API: `DELETE /api/admin/documents/[id]` — أرشفة لا حذف نهائي.
- [x] تخزين الملفات — Cloudinary مهيأ، يُفعّل عند الحاجة.
- [x] صلاحيات الوثائق — SUPER_ADMIN يرى الكل، PLATFORM_MANAGER يرى منصته.
- [x] منع الحذف النهائي — فقط تغيير الحالة إلى `ARCHIVED`.

### المرحلة 8 — تكامل الذكاء الاصطناعي (DeepSeek API) 🟢

> الهدف: استخدام DeepSeek كمساعد يقترح ويُسهّل، لا كحاكم يقرر. كل قرار يخص النقاط أو الاعتماد يبقى بشرياً؛ الذكاء الاصطناعي يُسرّع الصياغة والتحليل والاستعلام فقط.
> مبدأ أساسي: **AI-assist لا AI-decide** — أي مخرج من DeepSeek يُعرض كمسودة/اقتراح قابل للتعديل أو التجاهل، ولا يُكتب مباشرة في قاعدة البيانات دون مراجعة بشرية.
> **قيد الوصول (حاسم)**: كل ميزات الذكاء الاصطناعي مقصورة على دور `SUPER_ADMIN` (الإدارة العليا) فقط. لا يملك `PLATFORM_MANAGER` ولا `MEMBER` أي وصول لواجهات أو APIs الذكاء الاصطناعي — لا زر، لا رابط، لا استدعاء. يجب فرض هذا القيد على مستوى الـ API نفسه (فحص الدور قبل أي استدعاء DeepSeek) لا فقط بإخفاء الزر في الواجهة.

**أ — البنية التحتية الأساسية:**
- [x] إضافة متغيرات البيئة: `DEEPSEEK_API_KEY`, `DEEPSEEK_BASE_URL`, `DEEPSEEK_MODEL` (مثل `deepseek-chat`).
- [x] إنشاء `src/lib/ai/deepseek.ts` — عميل موحّد (DeepSeek متوافق مع OpenAI SDK، فيمكن استخدام `openai` package بـ `baseURL` مخصص).
- [x] إضافة دالة `chatComplete(prompt, system?, opts?)` عامة تُعاد استخدامها في كل الميزات أدناه.
- [x] **حارس صلاحية مركزي**: إضافة `requireSuperAdmin()` يُستدعى في بداية كل API من نوع `/api/admin/ai/*` ويرفض الطلب فوراً (403) إن لم يكن دور الجلسة `SUPER_ADMIN`.
- [x] **Rate limiting وCaching**: تقييد عدد الطلبات لكل مستخدم/دقيقة (إعادة استخدام `src/lib/rate-limit.ts` الموجود)، وتخزين مؤقت للمخرجات المتكررة (نفس الشهر/نفس العضو) لتقليل التكلفة.
- [x] **معالجة الأخطاء بأمان**: عند فشل أو بطء استدعاء DeepSeek، يجب أن تستمر الميزة الأساسية (تقرير، اعتماد، إدخال) بدون توقف — الذكاء الاصطناعي طبقة إضافية لا حرجة.
- [x] تسجيل استهلاك الـ API (عدد الطلبات، التكلفة التقديرية، هوية المسؤول المُستخدِم) في جدول `AiUsageLog` بسيط لمتابعة التكلفة والمساءلة.
- [x] عدم إرسال بيانات حساسة (بريد، هاتف) إلى الـ API — تمرير فقط ما يلزم للمهمة (أرقام، نقاط، أسماء عامة عند الحاجة).

**ب — التقارير الذكية بالذكاء الاصطناعي (المستوى 1: صياغة + المستوى 2: تحليل واتجاهات):**

> كل العناصر أدناه مرئية ومتاحة لـ `SUPER_ADMIN` فقط.

*المستوى 1 — صياغة الملخص التنفيذي:*
- [x] زر "توليد ملخص ذكي" في تبويب التقارير (`ReportsTab`) وصفحة `/admin/reports` يرسل الأرقام المجمّعة (الإجمالي، المتصدر، عدد النشطين، المنصة الأنشط) إلى DeepSeek ويطلب فقرة عربية صحفية قابلة للنشر مباشرة.
